# Go 失敗パターンカタログ

Go のコードを書く前・レビューする前に読む。形式は「症状 → ありがちな間違い → 正しい一手」。

**学習ループ**: カタログに無い想定外のミスが出たら、同じ形式で 1 エントリをこのファイルに追記する（ユーザー承認の上。dotfiles リポ側 `claude/failure-catalog/go.md` を編集すれば symlink 経由でライブ反映される）。

## 前提チェック（書き始める前）

- 言語機能・ライブラリを使う前に `go.mod` の Go バージョンと依存のバージョンを読む。記憶で判断せず、リポを読んで判断する
- パターン選択（エラーハンドリング・レイヤ分け・テストの書き方）の前に、隣接する既存ファイルを 2 つ読んで流儀を確認する

## goroutine 起動前の 3 質問

起動する前に 3 つ全部に答える。どれかの答えが「誰もいない」なら起動しない:

1. 誰がキャンセルするのか？（context は誰が閉じる？）
2. 誰が完了を待つのか？（WaitGroup / errgroup / チャネル受信者は？）
3. エラーはどこへ届くのか？（握りつぶしていないか？）

## カタログ

1. **エラーレスポンス後も処理が続き、二重レスポンス・panic・不正な後続処理**
   - 間違い: `http.Error()` やエラー用レスポンス送出の後に `return` を書き忘れる
   - 正しい一手: エラーレスポンスの直後に必ず `return` を置く

2. **DB コネクションが枯渇する／結果が欠けるのに気づかない**
   - 間違い: `rows.Close()` の defer 忘れ、ループ後の `rows.Err()` 未確認
   - 正しい一手: `defer rows.Close()` を Query 直後に置き、ループを抜けたら `rows.Err()` を確認する

3. **エラーパスでトランザクションが開きっぱなしになる**
   - 間違い: エラー分岐それぞれに `Rollback()` を書こうとして漏れる
   - 正しい一手: `Begin` 直後に `defer tx.Rollback()`（Commit 成功後の Rollback は no-op）。Commit のエラーも確認する

4. **`errors.Is`/`errors.As` が効かず、エラー種別分岐がすり抜ける**
   - 間違い: `fmt.Errorf("...: %v", err)` で包んでチェーンが切れる
   - 正しい一手: 包むときは `%w`。センチネル比較・型分岐がある層では特に必須

5. **nil map への書き込みで panic**
   - 間違い: `var m map[string]T` のまま代入する（構造体フィールドのゼロ値 map も同様）
   - 正しい一手: 書き込む map は `make` か複合リテラルで初期化してから使う

6. **ループ内 defer でリソースが関数終了まで解放されない（リーク・fd 枯渇）**
   - 間違い: `for` の中で `defer f.Close()`
   - 正しい一手: ループ本体を無名関数/別関数に切り出して defer するか、その場で明示的に Close する

7. **ループ内 `time.After` でタイマーがリークする**
   - 間違い: `select` を回すループの中で毎回 `time.After` を呼ぶ
   - 正しい一手: `time.NewTimer`/`time.NewTicker` をループ外で作り、`Stop()` する

8. **部分スライスへの append で元スライスのデータが壊れる**
   - 間違い: `s[:n]` など容量を共有したスライスに append して、元の後続要素を上書きする
   - 正しい一手: 独立させたいときは `copy` するか full slice expression（`s[:n:n]`）で容量を切る

9. **NULL になりうる列の Scan でエラー、またはゼロ値と NULL の区別が消える**
   - 間違い: NULL 許容カラムを `string`/`int` の素の型で受ける
   - 正しい一手: `sql.NullString` 等かポインタで受け、NULL の意味（未設定 vs 空）をアプリ側で明示する

10. **JSON の入出力でフィールドが落ちる・ゼロ値が消える**
    - 間違い: struct タグの書き漏れ・タイポ、`omitempty` が `0`/`false`/空文字まで落とすことの見落とし
    - 正しい一手: タグを隣接コードの流儀と突き合わせる。「ゼロ値も送りたい」フィールドに `omitempty` を付けない（区別が要るならポインタ）

11. **並行アクセスされる map / 共有状態で data race**
    - 間違い: 複数 goroutine から保護なしで map・スライス・カウンタを触る
    - 正しい一手: `sync.Mutex` 等で守る（用途が合えば `sync.Map`）。検出は `go test -race` を必ず通す

12. **リクエスト処理後も走る goroutine が途中で死ぬ**
    - 間違い: リクエストスコープの `ctx` をバックグラウンド処理へそのまま渡し、レスポンス返却と同時にキャンセルされる
    - 正しい一手: 処理の生存期間に合った context を選ぶ（値だけ引き継ぐなら `context.WithoutCancel`、独立寿命なら親を分ける）

13. **ループ変数のキャプチャで goroutine/クロージャが全部同じ値を見る**
    - 間違い: Go 1.22 未満のセマンティクスで `for _, v := range` の `v` をそのままクロージャに渡す
    - 正しい一手: まず `go.mod` の Go バージョンを確認（1.22 以降は per-iteration で解消）。未満ならループ内で変数を作り直すか引数で渡す

14. **時刻依存で flaky なテスト・日跨ぎ/タイムゾーンで壊れる処理**
    - 間違い: ロジック内で `time.Now()` を直接呼ぶ、TZ を暗黙に依存する
    - 正しい一手: 時刻取得を注入可能にする（周辺コードに既存の clock パターンがあればそれに合わせる）。TZ は明示する
