# 汎用レビューチェックリスト

言語・プロジェクトに依存しないレビュー観点。palmu-api-go のレビュー観点から汎用部分を抽出したもの。

**使い方**: 差分に該当する観点だけ適用する（例: DB 変更がなければ「データベース」「SQL」はスキップ）。プロジェクト固有の規約（coding-standards 等）がある場合はそちらを優先し、このリストは下敷きとして使う。

## 全般

- [ ] コードは他の人が最短時間で理解できるか
- [ ] 既存コードと一貫したスタイルで書かれているか
- [ ] 要件・タスクの目的を満たしているか。仕様とコードのロジックは整合しているか

## 凡ミス

- [ ] スペルミスはないか
- [ ] 未使用の変数・関数・無意味な処理が残っていないか
- [ ] デバッグ用コード（print 系の出力等）が残っていないか
- [ ] プロジェクトのテストがローカルで通っているか
- [ ] migration がある場合、可逆性（rollback）を確認したか

## 命名

- [ ] 明確で具体的な単語を選んでいるか。汎用的な名前（temp、data 等）を避けているか
- [ ] 名前は処理の説明ではなく、値や目的を表しているか
- [ ] 単数/複数を意識しているか
- [ ] bool 型は is/has/can/should を使い、否定形（disable 等）を避けているか
- [ ] 数値は count/amount や単位（seconds, days 等）がわかる名前か
- [ ] 時間・期間を扱う名前は期間がわかる接頭語（Current/Monthly/Daily 等）を付けているか

## 変数・関数

- [ ] 変数は不変にできているか。再代入していないか
- [ ] 変数のスコープは最小限か。使う直前に定義しているか
- [ ] グローバル変数・static 変数を使っていないか
- [ ] 関数は 50 行以下か。1 関数 1 責務になっているか
- [ ] 引数は少なくできているか。フラグ引数は関数分割を検討したか
- [ ] なるべく null を渡さない・返さない設計か
- [ ] エラーを握り潰していないか。呼び出し元へ明確に返しているか
- [ ] 現在時刻は関数の外から引数で渡して純粋関数にしているか

## クラス・構造

- [ ] ファイルは 800 行以下か。500 行超えたら分割を検討
- [ ] インスタンスが不正状態に陥らない設計になっているか
- [ ] 外部からアクセス不要なものを private にしているか
- [ ] 1 つの構造体・クラスが複数の責務を担っていないか
- [ ] 同じロジックが複数箇所にないか（ただし概念が違うなら共通化しない）
- [ ] 他のオブジェクトの内部状態を取得して呼び出し側で判断していないか（Tell, Don't Ask）
- [ ] 既存のユーティリティ・共通処理・ライブラリで代替できないか（車輪の再発明を避ける）

## 条件分岐・処理の流れ

- [ ] 早期リターンでネストを浅く（1-2 階層）できているか
- [ ] 条件式は肯定型でシンプルか。読みにくい条件式は説明変数を置いているか
- [ ] for 文の中で分離できる複数処理は縦に分割しているか
- [ ] 処理の過程で使う一時データをインスタンス変数にしていないか（引数・戻り値で渡す）

## コメント

- [ ] コメントがなくても理解できるコードになっているか（まずコードで表現する）
- [ ] コードからわからない背景・意図・欠陥のコメントは書いているか
- [ ] コードから自明なことをコメントしていないか
- [ ] コメントアウトされたコードを残していないか

## テスト

- [ ] 変更に対応するテストがあるか。命令網羅と境界値テストを書いているか
- [ ] テストケース名は「〜〜の場合、〜〜であること」を意識しているか
- [ ] フレーキーにならないか（時間依存・順序依存・外部リソース依存を避け、常に同じ結果を返す）
- [ ] テストの無効化・スキップをしていないか

## データベース（スキーマ変更がある場合）

- [ ] 正規化できているか。適切に index を貼っているか
- [ ] 外部キー制約を張っているか
- [ ] nullable を無駄につけていないか
- [ ] 金額等の正確な数値に FLOAT を使っていないか（DECIMAL を使う）
- [ ] ENUM 型・JSON 型・ポリモーフィックなカラムを安易に使っていないか
- [ ] ソフトデリート（deleted_at）は本当に必要か

## SQL・パフォーマンス

- [ ] N+1 になっていないか。ループ内で SQL 発行・通信をしていないか
- [ ] 適切な index が使われ、行数を絞れているか（必要なら EXPLAIN）
- [ ] 負荷のかかる SQL（サブクエリ、大量の in 句、or 句）を安易に使っていないか
- [ ] 実際の負荷を見積もったか。本番の 10 倍のレコード数でも問題ないか
- [ ] 途中で失敗すると不整合が発生する処理はトランザクションにできているか
- [ ] トランザクション処理を最小限にできているか（ロック待ち・デッドロック防止）

## バッチ・非同期処理（該当する場合）

- [ ] タイムアウトや失敗を前提とした設計か。失敗時に検知できるか
- [ ] 冪等性があるか。再実行可能か（長い処理はステップ化）
- [ ] 可能であれば差分更新にしているか

## API（該当する場合）

- [ ] 仕様を正確に共有できるか（成功/失敗の各パターン、データの型と定義）
- [ ] REST を意識しているか。異なるコンテキストを無理に 1 つの endpoint にまとめていないか
- [ ] パラメーター・レスポンスの命名は既存 API と一貫しているか
- [ ] API の振る舞いが変わった場合、クライアント（利用側）への影響を確認したか

## 非正常系

- [ ] 認証切れ・未ログイン等の異常系を考慮しているか
- [ ] データ量・アクセスが 10 倍になっても大丈夫か
- [ ] 外部サービスやミドルウェアが落ちているときの考慮はあるか
- [ ] デプロイ前後・新旧バージョン共存時のリスクを考慮したか（後方互換性）
- [ ] クライアント側なら: 連打・通信中の操作、長い文字列・0 件・桁数超過の考慮

## 影響範囲

- [ ] 変更した関数・変数・DB の利用箇所を全て確認したか
- [ ] データ定義の変更時、それを参照する外部システム（分析基盤・集計・帳票等）への影響を確認したか

## セキュリティ・設定

- [ ] 接続情報や認証鍵をリポジトリに含めていないか（環境変数等で管理）

## PR の作り方

- [ ] 機能追加とリファクタリングを同時にやっていないか
- [ ] PR が大きすぎないか。分割できないか
- [ ] マイナーライブラリを無駄に導入していないか
